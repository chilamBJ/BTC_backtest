# Dashboard「加载失败: 网络请求失败」根因与修复说明

## 现象

- 切换轮次或刷新后，偶发出现：**加载失败: 网络请求失败，请刷新重试**（可稍后自动重试或手动刷新）。
- 删除第 1 批次后仍会偶发，与「某一批次数据错误」无直接关系。

## 根因（诊断结论）

通过本机 → 云端公网连续请求 Dashboard API 的测试（`scripts/diagnose_dashboard_network.py`）得到：

1. **IncompleteRead**  
   客户端有时只收到响应体的一部分，剩余字节未到达即连接结束，导致 `Failed to fetch` / 解析 JSON 失败。  
   → 说明在**公网链路上，较大或稍长的响应更容易被截断**（中间设备、NAT、不稳定网络等）。

2. **503 Service Unavailable**  
   偶发 503，与并发、超时或中间代理有关，进一步说明**公网访问同一端口不稳定**。

3. **小响应 vs 大响应**  
   - 小响应（如 `/api/health` 几十字节）也有少量失败（IncompleteRead/503）。  
   - 大响应（`/api/data` 数千字节）失败率更高。  
   → 结论：**单次响应体积越大，越容易在公网链路上被截断或超时**；同时存在**整体链路不稳定**因素。

因此，根因是：**公网到云端的 HTTP 连接不稳定 + 单次响应体积偏大**，而不是「某一批次数据错误」或单纯服务端逻辑错误。

## 已做修复

1. **缩小单次响应体积**
   - 后端：对 `per_page` / `log_per_page` 做上限（例如 20），避免单次返回过多条。
   - 前端：默认每页 12 条（原 30），降低单次 `/api/data` 的 JSON 体积，减少被截断概率。

2. **加强前端重试**
   - 重试次数提高到 5 次，退避间隔：1.5s → 2.5s → 4s → 6s → 9s。
   - 失败时提示「X 秒后自动重试 (当前次/总次数)」，减少用户反复手动刷新。

3. **诊断与调试**
   - 新增 `/api/debug/small`（极小 JSON），便于对比「小响应 vs 大响应」失败率。
   - 脚本：`python scripts/diagnose_dashboard_network.py [--base URL] [--rounds N]`，用于复现和统计成功率。

## 若仍偶发失败

- **优先建议**：使用 **SSH 隧道** 访问 Dashboard，避免走公网 8080，链路最稳。  
  - 例：`ssh -L 18080:127.0.0.1:8080 root@<云主机>`，浏览器打开 `http://127.0.0.1:18080`。
- 若必须直连公网：保持当前「小分页 + 多次重试」即可；极端不稳定时可考虑在云端加反向代理（如 nginx）或 HTTPS，由运维侧进一步优化链路。
